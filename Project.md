# Minecraft 闪连 (MC-FlashLink V2) 系统开发需求与架构设计说明书（虚拟网卡版）

## 1. 项目概述

### 1.1 项目简介
**MC-FlashLink V2** 是一款基于**虚拟局域网（VNI）**技术的《我的世界》（Minecraft）极简联机工具。系统放弃了传统的应用层 TCP 端口转发模式，改为在用户电脑上创建**虚拟网卡（TUN 驱动）**。

通过构建异地虚拟局域网，系统为每个加入房间的玩家分配一个同网段的虚拟 IP（例如 `10.26.0.X`）。玩家之间如同处于同一个 Wi-Fi 下，完美支持 Minecraft Java 版（TCP）与基岩版（UDP），且无需手动输入端口，打开游戏即可在局域网列表看到好友。

### 1.2 核心业务流程
1. **房主创建房间**：房主点击“创建房间”，客户端连接中心服务器，服务器为其分配房间号，并在房主电脑的虚拟网卡上配置 IP `10.26.0.1`。
2. **客星加入房间**：客星输入房间号点击“加入”，服务器验证后分配 IP `10.26.0.2`。
3. **建立数据隧道**：房主与客星的客户端建立 UDP 通道（优先尝试 P2P 打洞，失败则通过服务器中转）。
4. **游戏联机**：房主在游戏里“开放局域网”，客星游戏内的网络协议栈会通过虚拟网卡自动发现 `10.26.0.1` 广播的游戏包，实现一键加入。

## 2. 技术栈选型 (Technology Stack)

由于涉及操作系统底层的虚拟网卡（TUN）操作，纯 Java 难以直接调用，需要结合 JNI/JNA 或外部轻量级组件：

* **中心服务端 (Backend)**：
  * **核心框架**：Spring Boot 2.7.x
  * **核心职责**：管理房间、分配虚拟 IP（类似 DHCP 控制器）、协助客户端进行 STUN/P2P 打洞握手。
* **客户端 (Client)**：
  * **UI 框架**：JavaFX 17+
  * **底层驱动操作**：通过 Java (JNA) 调用 Windows 的 **Wintun** 驱动（或集成开源的 WireGuard/tinc 核心，Java 作为外壳控制台）。
  * **网络数据通道**：Java NIO / Netty (仅用于传输封装后的三层 IP 数据包)。

## 3. 核心模块详细设计

### 3.1 虚拟 IP 地址管理 (IPAM 模块)
服务端需要维护一个私有网段（例如 `10.26.0.0/24`），并为每个房间划分独立的子网或动态分配。
* **房主默认分配**：`10.26.x.1`
* **客星依次分配**：`10.26.x.2`, `10.26.x.3` ...

### 3.2 虚拟网卡与隧道工作原理 (TUN & Tunnel)
[ 玩家 A (10.26.0.1) ]                       [ 玩家 B (10.26.0.2) ]
Minecraft 游戏                               Minecraft 游戏
│ (发送本地局域网包)                         ▲
▼                                           │ (收到解包后的三层IP包)
[ 虚拟网卡 Wintun ]                           [ 虚拟网卡 Wintun ]
│ (拦截到 IP 报文)                           ▲
▼                                           │
[ JavaFX 客户端 ]                            [ JavaFX 客户端 ]
│                                           ▲
└───────► [ 物理网卡: UDP 隧道加密包 ] ──────┘
(优先 P2P 直连 / 服务器中转)

1. **流量拦截**：当 MC 游戏发送数据到 `10.26.0.2` 时，Windows 系统会将流量路由到 Wintun 虚拟网卡。
2. **读取报文**：Java 客户端从 Wintun 驱动的文件句柄（Handle）中，读取出原始的 **三层 IP 数据报文（Layer 3 IP Packet）**。
3. **隧道传输**：客户端将该 IP 报文作为 Payload，包裹在自定义的 UDP 数据包中，通过公网直接（或中转）发给目标玩家的客户端。
4. **注入网卡**：目标玩家的 Java 客户端收到 UDP 包，解出里面的原始 IP 报文，写入本地的 Wintun 驱动，系统网络栈接收，MC 游戏成功收到数据。

### 3.3 网络穿透策略 (P2P & STUN)
为了降低开发团队自行编写组网底层的难度，建议 CodeX 团队参考以下两种实现路线：
* **路线 A（自研轻量级）**：Java 客户端使用 Java UDP Socket，配合服务端进行 STUN 打洞。打洞成功则 P2P 传输网卡数据，失败则走 Spring Boot 服务器中转。
* **路线 B（集成成熟开源，强烈推荐）**：Java 只做外壳和 UI，底层直接封装 **WireGuard**（通过配置生成的 Conf 文件后台启动 wg-quick）或 **Tailscale/Derp**。这种方式稳定性极高，安全且自带极强的打洞能力。

## 4. 开发阶段与交付物规划 (Milestones)

### 阶段一：环境调研与虚拟网卡跑通 (环境交付)
* **目标**：在 Windows 环境下通过 Java (JNA) 成功加载并启用 Wintun 驱动，实现给网卡成功绑定 IP，并能通过 Java 代码读取到网卡流量。
* **交付物**：Wintun 驱动调用 Demo（Java 版）。

### 阶段二：中心控制器与控制流 (拓扑交付)
* **目标**：Spring Boot 实现房间管理与虚拟 IP 分配算法。客户端连接后，能自动获取 IP 并正确配置到本地虚拟网卡上。此时两台电脑各自拥有了 `10.26.0.X` 的身份，但还不能互通。
* **交付物**：IPAM 控制器源码、网卡自动配置模块。

### 阶段三：UDP 数据隧道与组网 (组网交付)
* **目标**：实现客户端之间的 UDP 监听与转发，完成“网卡 A -> 客户端 A -> UDP 隧道 -> 客户端 B -> 网卡 B”的完整闭环。在两台异地电脑上实现 `ping 10.26.0.1` 成功。
* **交付物**：全功能联机组网引擎。

### 阶段四：UI 整合与权限优化 (发布版交付)
* **目标**：整合 JavaFX 界面。**注意：由于安装和操作虚拟网卡需要系统管理员权限，客户端在启动时必须提示请求 UAC（以管理员身份运行）。**
* **交付物**：带自签名（或提示管理员权限）的客户端安装包、服务端部署镜像。

## 5. 开发注意事项与潜在风险提醒

1. **管理员权限 (UAC)**：操作虚拟网卡（创建、删改 IP、修改路由表）**必须**拥有 Windows 管理员权限。Java 客户端打包时必须配置清单（Manifest）以强制请求管理员权限启动。
2. **防火墙阻挡**：Windows 防火墙默认会拦截非信任网卡的 UDP 广播或未知公网端口。开发团队需要提供一键调用 `netsh advfirewall` 命令来允许软件通过发射墙，否则玩家可能“能 Ping 通，但游戏里找不到房间”。
3. **MTU 调优**：由于在 IP 报文外又包裹了一层 UDP 隧道头，必须将虚拟网卡的 MTU 设置得比物理网卡小（建议设为 `1420` 或 `1380`），否则大封包会导致 IP 分片，造成游戏极度卡顿或丢包。
